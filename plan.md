## Планирование

### Перечень автоматизируемых сценариев
1. Корректное отображение формы при выборе типа покупки
2. Ввод валидных данных в поля формы
3. Ввод невалидных данных в поля формы
4. Очистка полей формы
5. Отправка валидных данных
6. Отправка невалидных данных

### Перечень используемых инструментов с обоснованием выбора
1. Docker - избегаем локальной установки и настройки всех компонентов необходимых для тестирования (СУБД, Node.js, OpenJDK).
2. JUnit5 - наиболее часто используемая платформа для написания авто тестов. Лучше остальных (JUnit4, TestNG) мной изученная библиотека.
3. Gradle - инструмент автоматизации сборки и удобного управления зависимостями. Более гибкий и менее многословный за счёт использования Groovy, по сравнению с Maven. Есть встроенный функционал отчётности.
4. Selenide - фреймворк для автоматизированного тестирования веб-приложений. Простой и мощный инструмент.
5. Java - ЯП для написания тестов. Удобен в использовании с вышеперечисленными инструментами.

### Перечень и описание возможных рисков при автоматизации
1. При неграмотном подходе и отсутствии опыта есть риск, что разработка авто тестов может превратиться в процесс создания приложения для проверки сервиса и серьезно затянуться.
2. Автоматические тесты выполняются строго по сценарию, в то время как ручном тестировании человек обращает внимание на сторонние детали и может найти неожиданные ошибки

### Интервальная оценка с учётом рисков (в часах) 
1. Подготовка сборки для контейнеризации сервиса в Docker и описание процедуры запуска/перезапуска сборки (3 часа)
2. Написание и прогон авто тестов по перечисленным сценариям (20 часов)
3. Формирование отчётных документов по итогам тестирования (Issue, Reports) (10 часов)
4. Формирование отчёта по итогам автоматизации (5 часов)
